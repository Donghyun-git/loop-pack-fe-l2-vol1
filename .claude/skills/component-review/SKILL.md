# Component Review Skill

> "파일을 쪼갰냐"가 아니라 **"한 번에 한 가지만 읽을 수 있냐"** 를 본다.
> 모든 분리는 _"이게 무엇을 좋게 하나"_ 를 한 문장으로 답할 수 있어야 한다. 답이 없으면 분리하지 않는다.
> 큰 컴포넌트가 큰 Hook으로 옮겨가는 단순 이관이라면 분리하지 않는다.

## 리뷰 순서

1. 상태 3분할 → 2. 레이어 경계 → 3. Hook 역할 → 4. 분리/비분리 판단 → 5. 의존 방향 → 6. Effect 정당성 → 7. 합성/Props → 8. 제어 흐름 → 최종 리트머스

---

## 1. 상태 3분할 (모든 판단의 기반)

- 상태를 **서버 / 클라이언트 / 파생값** 으로 분류했나?
- 🚩 **파생 가능한 값을 `useState` + `useEffect`로 동기화** → 계산으로 바꿔라 (예: `totalPages`, 필터링된 목록, 뱃지 플래그).
- 🚩 서버 상태를 컴포넌트가 직접 `useState`+`fetch`로 관리 → 서버 상태 훅으로.
- 상태의 **소속은 그것을 쓰는 모두의 최소 공통 조상**. 하위 컴포넌트가 훅을 각자 호출하면 상태가 공유되지 않고 복제된다.

## 2. 레이어 경계 (UI / Hook / Service / Utils)

- 🚩 UI(JSX) 파일에 `fetch`/`axios` 직접 호출 → **Service로** (DIP: 훅은 service 함수에만 의존).
- 🚩 포맷/도메인 규칙 순수함수가 컴포넌트/렌더 안에 인라인 → **Utils로**.
- 컴포넌트가 서버 응답의 세부 구조를 너무 많이 아는가? → 훅/서비스가 흡수.
- 판별: **Service만 봐도 API 스펙(endpoint/DTO)이, Hook만 봐도 기능 흐름이, 컴포넌트만 봐도 화면이** 보이는가?

## 3. Hook 역할

- **한 문장**으로 설명되나? 설명에 "그리고"가 두 번 들어가면 분리 후보.
- **내부에서 React 훅(`useState`/`useEffect`…)을 쓰면 반드시 훅** (반환이 `void`여도 OK). 안 쓰면 훅이 아니라 순수 util → `use` 프리픽스 금지.
- 🚩 `useData`처럼 모호한 이름 금지 — 역할이 이름에 드러나야 (`useProductFilters`).
- 🚩 순수 상태 훅에 `afterX` 같은 **부수효과 콜백을 꿰지 마라** (SRP 위반 + 테스트 오염). 결합은 상위 조합 레이어에서 나란히 호출로.
- Hook이 JSX/스타일을 반환하지 않나?

## 4. 분리 / 비분리 판단

**분리한다** (하나라도 해당):

- 상태 유형이 다르다 (서버/클라이언트/파생)
- 순수함수 + 재사용/테스트 가치 → util
- 함께 변하는 것끼리 응집이 높다 → 하나로 묶어 분리
- JSX 반환 표현 로직 → 컴포넌트
- 비자명한 코드가 **3회 이상** 반복 → primitive

**분리하지 않는다** (하나라도 해당):

- 🚩 **UI 위치/상호작용 방식만 다르다** — 같은 관심사면 상태는 안 나눔 (검색 input·정렬 select이 따로 그려진다고 상태를 쪼개지 마라).
- 🚩 **분리해도 결합이 위로 옮겨갈 뿐** — 본질적 결합은 못 없앤다. 배선만 늘면 합쳐 둬라 (우연한 중복 ≠ 본질적 결합).
- 파생 가능한 값 → 계산 (state로 분리 X)
- 트리비얼(한 줄) + 재사용/테스트 이득 없음
- 2회 반복 → primitive화 보류(3회부터)

## 5. 의존 방향 & 타입 배치

- 🚩 **util → service 역방향 import** 금지. 공유 타입은 둘 다 아래로 참조하는 **leaf**에 둔다.
- 🚩 순환 참조 (A가 B의 함수를, B가 A의 타입을) → 타입을 중립 위치로.
- **도메인 모델을 전송 계층(`services/`) 밑에 두지 마라** — 도메인은 상위 개념.
- **DTO(엔드포인트 요청/응답)** 와 **도메인 모델(엔티티 본질 속성)** 을 구분. 리터럴 유니온이라고 다 도메인 아님 (정렬 옵션=DTO, 카테고리=도메인).

## 6. useEffect 정당성

- 남은 `useEffect`마다 묻는다: **"외부 시스템과의 동기화인가?"**
- 🚩 사용자 이벤트의 결과 → Effect 말고 **이벤트 핸들러**에서.
- 🚩 다른 state로부터 계산 가능 → Effect 말고 **렌더 중 계산**.
- 정당한 예: `scrollTo`, localStorage 동기화, URL 동기화, 외부 구독.

## 7. 합성 / Props

- 🚩 props가 **5개 초과** → 재검토. 새 prop 전에 **합성(children/slot)** 으로 풀 수 있나?
- boolean prop은 긍정형 (`isOpen`, `hasError`).
- 조건부 props → optional보다 **discriminated union**, 더 복잡해진다면 namespace 분리
- Props Drilling 3단계 초과 → Context 검토 (그 전엔 과함).

## 8. 제어 흐름 / 패턴매칭

- 🚩 얽힌 boolean 파생값이 **상호배타 상태**라면 → 단일 상태 + 패턴매칭(`getXStatus` + 룩업/switch).
- 단, **동시에 켜지는 독립 플래그**(뱃지 여러 개)는 패턴매칭 대상 아님 → 규칙 리스트/개별 술어 유지.
- 조건부 렌더링은 early return으로.

## 최종 검증 결과

각 분리에 대해 답하라: **"이 분리가 무엇을 좋게 하나?"** — 한 문장으로 못 하면 되돌려라.
검증 질문: _"이 조건 하나(예: 정렬 옵션) 바꾸려고 몇 파일/몇 줄을 읽어야 하나?"_ 국소적이면 분리가 값을 한 것이다.

## 리뷰 출력 형식

아래 4개 섹션을 모두 포함한다. 해당 항목이 없으면 그 섹션에 "없음"이라고 명시한다.

- 🚩 문제: `위치 · 어떤 원칙 위반 · 한 줄 근거 · 제안`
- 🐞 동작 결함: 구조 리뷰 중 발견한 기능/런타임 문제 — 무반응 액션(요청 후 화면 미갱신), 에러 미처리(catch/res.ok 부재), 크래시 가능성(undefined 접근). 구조가 아니라는 이유로 생략하지 않는다.
- ✅ 잘된 분리: 한 문장 근거와 함께 인정
- ⚖️ 판단 경계(예: 2회 반복, 얇은 컴포넌트): 단정 말고 트레이드오프 제시 후 권고
